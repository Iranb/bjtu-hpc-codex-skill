# Guardrails

Read this file when changing policy, submitting jobs, touching credentials, sharing datasets, or diagnosing risky queue/resource behavior.

## Guardrails

- Use the token file `~/.bjtu_hpc_token`; never hardcode tokens or certificate values.
- For multi-account work, treat `~/.bjtu_hpc_accounts.json` as the auth source of truth and `~/.bjtu_hpc_token` only as a legacy compatibility cache. Prefer `--auth-account NAME` on common commands.
- For evidence-producing GPU training submissions, default to one native packed Slurm job per launch unit, but never submit it before a pre-submit runability gate. Use the monitor resource snapshot first and start ordinary jobs at `1GPU/6CPU`: packed pairs use `2GPU/12CPU` (`--nodes=1 --ntasks=2 --cpus-per-task=6 --gres=gpu:2 --gres-flags=disable-binding`), singletons use `1GPU/6CPU`, and wide jobs use `N GPU / 6N CPU` when enough independent one-GPU children exist. Fall back to `1GPU/4CPU` shapes (`2GPU/8CPU`, `1GPU/4CPU`, or wide `--cpus-per-task=4`) only when exact-script `sbatch --test-only` cannot run the 1:6 shape directly or reports/demonstrates `Resources`, reservation, same-node CPU pressure, or GPU/GRES shape pressure; do not lower CPU for pure `Priority`, dependency holds, or `QOSMaxJobsPerUserLimit`. CPU-rich `1:8`, `1:12`, or `1:16` shapes are optional only when the user explicitly asks for CPU-rich work or snapshot plus test-only proves immediate start without reducing GPU occupancy. If a same node has many free GPUs but limited CPUs, allow one wide allocation up to `8GPU`, for example `--ntasks=8 --cpus-per-task=4 --gres=gpu:8`, only when `hpc_resource_planner.py --available-children N` or `--child-manifest` proves there are at least `N` independent one-GPU children and `hpc_native_sbatch_builder.py` has generated a matching N-child script. If each child has observed or strongly bounded peak VRAM below `16GB` on BJTU V100-32GB, allow a low-memory GPU-sharing allocation that maps at most two child processes to each allocated GPU, such as `--ntasks=2N --gres=gpu:N`; never run more than two code executions on one GPU and never use this mode when VRAM is unknown or close to `16GB`. If the packed 2GPU shape still cannot be scheduled and native evidence shows 1GPU can run, split the affected single-GPU-capable experiment pair into native `1GPU/6CPU` singleton jobs and test `1GPU/4CPU` only when the 6-CPU singleton cannot run directly. If the user explicitly prioritizes GPU utilization and same-node Slurm evidence supports GPU-fill fragment mode, submit one allocation for the whole fragment, such as `--ntasks=4 --cpus-per-task=2 --gres=gpu:4`, with one independent single-GPU child per task. Each child experiment must be bound to exactly one Slurm-allocated `CUDA_VISIBLE_DEVICES` entry and matching CPU threads. Do not go below `6` CPUs per child training task for ordinary packed or singleton jobs, below `4` CPUs per child for explicitly recorded resource-wait packed or 1GPU compatibility fallbacks, or below `2` CPUs per child for explicitly recorded GPU-fill fragment jobs. Keep historical higher-CPU jobs in queue unless the user explicitly authorizes replacement; after authorization, replace only current-project `PENDING` jobs that are still in `Resources`, reservation pressure, GPU/GRES scarcity, or 2GPU resource-shape pressure and whose child experiment labels/parameters will be preserved in the reduced or split replacement. Use native `sbatch` for all GPU training submissions whose results may become experiment evidence. Do not rely on the portal PyTorch-GPU app to honor `--cpus-per-task` or `--gres-flags`; it has been observed to silently generate a Slurm script with only `1` CPU despite a multi-CPU portal payload. A direct portal-app `1CPU/1GPU` training allocation is forbidden for evidence-producing runs.
- If a GPU training job is discovered with `NumCPUs=1`, `CPUs/Task=1`, and `gres/gpu=1`, classify it as wrong-shape immediately. Sync lightweight logs, record the failure/replacement lineage, and replace it using native `sbatch` with verified CPU/GRES allocation. Cancel the wrong-shape job only when it is the same experiment/run being replaced or the user explicitly authorizes cancellation; never cancel unrelated jobs.
- Every GPU submit path that claims CPU/GRES correctness must run native verification using the real Slurm job id. In `--wait` flows, check both the immediate `job` row and `wait.job`; do not skip allocation verification just because the immediate portal row lacked a job id. If the MCP submit-and-verify wrapper has not been patched to read `wait.job`, perform a manual `scontrol show job <job_id>` verification before reporting success.
- For experiment batches, cap each saved auth account at two run-slot packed jobs plus two queued packed follow-up jobs. Current Slurm `QOSMaxJobsPerUserLimit` behavior caps each account at four non-terminal jobs total. Each packed job contains two child experiments, so the default cap is four packed jobs or eight child experiments per account when resources and submit limits allow it. When a specific auth account is selected, check its current non-terminal packed jobs and submit enough new packed jobs to fill that account to four total non-terminal packed jobs, limited only by available experiment pairs and submit rejections. Do not stop after one successful packed job when open slots remain.
- When experiments are being submitted one at a time, do not precompute a multi-job virtual packing plan and then submit it unchanged. Prefer `python3 hpc_plan_from_snapshot.py --planner-json` before each submit so one live `hpc_queue_summary.py --json --jobs 4` snapshot feeds exactly one planner run through `--queue-json`; alternatively run the two commands explicitly. Use only `next_action`, submit and verify that one job, then refresh queue/resources and run the snapshot-backed planner again. This keeps account caps, pending jobs, QOS limits, and same-node CPU/GPU fragments current without duplicating live queue sweeps.
- When no same-node candidate fits, planner `queue_probe` output is backlog guidance, not an immediate submit recommendation. Plain `queue_probe` must have `do_not_submit=true` and `totals.submissions_to_do_now=0`; use `--allow-queued-submit` only when intentionally creating queued backlog and after exact-script `sbatch --test-only` evidence. Do not submit high-CPU `2GPU/16CPU`, `2GPU/24CPU`, or `2GPU/32CPU` jobs solely from a no-fit `queue_probe`; prefer ordinary `2GPU/12CPU` backlog when it has plausible start evidence, or `2GPU/8CPU` when 1:6 would wait for `Resources`, reservation, or same-node CPU pressure. If a pending job is already fallback `2GPU/8CPU` and the visible free GPUs have too little same-node CPU even for GPU-fill, preserve queue position and report same-node CPU exhaustion.
- If `hpc_resource_planner.py` reports no `next_action` but free cluster GPUs remain, inspect `cluster_diagnostics` and account `diagnostics`. `slot_fragmentation` means the account is job-slot full but not GPU-full because one or more running slots are 1GPU singletons. Do not submit beyond the configured account cap as a workaround; let running singletons finish and make the next refill packed, wide, or GPU-fill. `dependency_held_followups` means dependencies are limiting refill flexibility; use dependencies only when strict ordering is required.
- Current validated scheduling policy: each saved auth account, for example `account_a` through `account_e`, can carry two run-slot jobs plus two queued follow-ups when its token is valid and scheduler submit limits allow it.
- If strict refill ordering is required, queue packed follow-up jobs with native Slurm dependencies such as `--dependency=afterany:<job_id>`; otherwise prefer plain queued follow-ups so Slurm/QOS can start the next eligible packed job as soon as a running slot opens.
- Native single-GPU training jobs are exceptions, not the default. Use them only when the user explicitly requests a singleton, a repair must preserve a singleton run, the packed 2GPU shape cannot be scheduled while a 1GPU singleton can run, or WorkflowGuard records why packed launch is unsafe. Single-GPU compatibility jobs must request `--ntasks=1 --cpus-per-task=6 --gres=gpu:1 --gres-flags=disable-binding` by default, may fall back to `--cpus-per-task=4` only for resource-wait evidence, and must be recorded as exceptions with the skipped 2GPU reason. Because a singleton consumes a full user running-job slot while using only one GPU, prefer pairing future independent singleton children into one packed/wide allocation whenever the pre-submit gate can schedule it. Do not automatically downgrade a true multi-GPU/DDP experiment that needs two GPUs in the same process.
- If queued follow-up submissions hit `QOSMaxSubmitJobPerUserLimit` or a similar submit cap, record them in the local launch plan and submit later when a slot clears rather than retrying in a loop.
- Multi-account launches must keep account-local code, outputs, and environments under the corresponding cluster OS home. Shared datasets can cross accounts by ACL, but runtime paths should not cross accounts.
- Prefer the portal proxy from `hpc_winscp_info.py` for SSH/SFTP.
- Prefer a job-side probe over trying to infer GPU/CPU details from the login machine.
- Before re-uploading a dataset for another account, first test whether the target cluster OS user can reuse the existing cluster path via Unix permissions or ACLs. Portal tokens and the Web file-share UI do not by themselves grant Slurm jobs read access to another account's files.
- For cross-account dataset reuse, use the minimum permission that works: execute-only ACL on the source home directory when the dataset subtree is already readable, recursive read-only ACLs only when the subtree blocks reads.
- After ACL changes, verify reads as the target cluster OS user and pass the shared absolute path or a target-home symlink into training configs; do not duplicate large datasets unless access cannot be made safe.
- Datasets may be shared by ACL, but Python/conda runtime environments must live under the account that runs the job. Do not point `<cluster_account>` jobs at `/data/home/<cluster_account>/envs/...`; copy or rebuild the environment under `/data/home/<cluster_account>/envs/...`.
- For conda environment copies, use `conda create --copy --clone` as the target cluster OS user and verify owner plus inode samples. Avoid default clone hardlinks when the intent is an account-local copy.
- Use `dataset_upload_progress.py` before restarting dataset transfers; it detects completed files and active `.part` files by size.
- For resumable archive uploads, treat cluster-side `.part` size as the progress source of truth when source-side state/log SSH reads hang.
- Do not restart a resumable upload screen just because source-side state reads fail. First compare cluster-side `.part` size twice, 10-30 seconds apart, to determine whether bytes are still increasing.
- For dataset transfer, do not assume OpenSSH `scp` works from the source server. It authenticated to `<proxy_host>:<proxy_port>` but exited with status `255`; source-side Paramiko/SFTP works through the same proxy.
- Portal web upload API from a source host was tested with `hpc_upload.py` and was much slower than resumable SFTP-style transfer. Do not treat the web upload API as a faster dataset-transfer path unless retested.
- If upload or query APIs return auth errors, refresh the token once and retry.
- Read `references/hpc_workflow.md` when you need validated platform results and current environment notes.
